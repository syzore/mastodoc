# Documentation Architecture & Guidelines

## 1. The Core Philosophy
This documentation system is designed to be maintained primarily by AI coding agents. Code and documentation are one. A feature or refactor is not complete until the relevant documentation has been updated to reflect the new codebase reality.

## 2. The 3-Level Tree Structure
This project strictly enforces a 3-level hierarchical documentation tree. You MUST maintain this exact structure and never nest files deeper than this.

### Level 1: The Root (Master Docs)
*   **Location:** `docs/README.md`
*   **Purpose:** The entry point. It contains a high-level summary of the project and an index of links pointing directly to the Level 2 Branches.

### Level 2: The Branches (Feature Docs)
*   **Location:** `docs/features/*.md` (e.g., `auth.md`, `gamification.md`)
*   **Purpose:** Dedicated markdown files for each major feature or architectural domain. 
*   **Rule:** These files do not contain sub-folders. They exist flatly inside the `features/` directory.

### Level 3: The Leaves (Code Links & Logic)
*   **Location:** Inside the Level 2 Feature markdown files.
*   **Purpose:** The actual content of the branches. 
*   **Rule:** Leaves must contain relative markdown links pointing directly to the source code files (e.g., `[Auth Service](../../lib/services/auth.dart)`). NEVER use static line numbers, as they drift over time.

## 3. AI Maintenance Protocol
When asked to update the codebase, you must automatically execute this checklist before finishing:
1.  **Identify Drift:** Determine which Level 2 Feature Branch governs the code you just changed.
2.  **Update the Leaves:** Modify the logic descriptions and update any file path links if you renamed or moved files.
3.  **Update the Root:** If you built a completely new feature, create a new Level 2 file and add its link to the Level 1 Master Doc.
